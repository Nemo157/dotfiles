---
description: Sync the second brain — fill journal gaps from GitHub/Linear/Calendar/Slack, generate rollups, audit tasks
agent: brain
---

Synchronize the second brain at `~/.local/share/second-brain/` by detecting gaps in the journal, generating rollups, and auditing tasks.

## Workflow

Run these steps in order:

### 1. Check Brain State

```bash
ls ~/.local/share/second-brain/journal/daily/ | tail -14
```

Identify the last 7 days (from today backwards). Note which days have entries and which are missing.

### 2. Read Sync State

Record the current UTC timestamp **before** running any queries — this becomes the "last synced at" value written at the end:

```bash
SYNC_TIMESTAMP=$(date -u +%Y-%m-%dT%H:%M:%SZ)
```

Then read the existing sync state file:

```bash
cat ~/.local/share/second-brain/.brain-sync.json 2>/dev/null || echo '{}'
```

Extract `github.last_query_at`, `linear.last_query_at`, and `slack.last_query_at` if present. Use `github.last_query_at` as `$START_DATE` for GitHub queries, `linear.last_query_at` as `$LINEAR_START_DATE` for Linear queries, and `slack.last_query_at` as `$SLACK_START_DATE` for Slack queries. If any value is missing, fall back to the earliest missing day from step 1 (or 7 days ago if all days have entries).

### 3. Discover GitHub Activity

Query GitHub for all PRs and issues where `Nemo157` was involved over the sync period. Add an `--owner` flag to scope queries to the relevant GitHub organization (see profile-specific configuration appended to this command).

```bash
# PRs involved in (covers authored, reviewed, commented, mentioned)
gh search prs --involves=Nemo157 --updated='>=$START_DATE' \
  --json number,title,repository,url,state,updatedAt,createdAt

# Issues involved in
gh search issues --involves=Nemo157 --updated='>=$START_DATE' \
  --json number,title,repository,url,state,updatedAt,createdAt
```

### 4. Inspect Each PR/Issue for Interaction Timestamps

For each PR/issue found, fetch the detailed interaction history using the wrapper scripts:

```bash
# For PRs — returns metadata, reviews, review_comments, and commits filtered to the current user
gh-pr-activity OWNER/REPO PR_NUMBER

# For issues — returns metadata and comments filtered to the current user
gh-issue-activity OWNER/REPO ISSUE_NUMBER
```

Both scripts output a single JSON object. `gh-pr-activity` returns `{metadata, reviews, review_comments, commits}` and `gh-issue-activity` returns `{metadata, comments}`.

From these, build a list of interaction events with timestamps:
- **Authored PR**: use the PR `created_at`
- **Reviewed PR**: use each review's `submitted_at`
- **Commented**: use each comment's `created_at`
- **Committed**: use the commit date
- **Merged own PR**: use `merged_at` if the user is the author

Group these events by date (in local time) to assign them to the correct daily journal.

### 5. Discover Calendar Events

Use the Google Calendar MCP tools to find meetings for each day in the sync period:

- Use `list-calendars` to identify the work calendar
- Use `list-events` for each day in the sync period, filtering to the work calendar
- For each meeting: extract time, title, attendees

**Attendee formatting:**
- If attendees are only mailing lists or group addresses, list participants as "dev team" instead of expanding the list names
- If a meeting has both mailing lists and named individuals, use "dev team" for the lists and list the named individuals separately
- Only list named individuals when they are explicitly in the attendee list (not behind a mailing list)

Generate `### HH:MM (auto-generated by brain-sync)` entries with `#auto-sync` tag:

```markdown
### HH:MM (auto-generated by brain-sync)

Meeting: **title** with dev team

#auto-sync
```

**Reconcile moved/cancelled meetings:**

After generating new calendar entries, scan existing daily entries for the sync period that contain `Meeting:` lines with `#auto-sync`. For each such entry, check whether a matching calendar event still exists at that time and date (match on title). If an auto-synced meeting entry exists in the journal but the corresponding event is no longer on the calendar, update the entry to mark it as moved/cancelled:

```markdown
### HH:MM (auto-generated by brain-sync)

~~Meeting: **title** with attendees~~ — moved/cancelled (no longer on calendar as of YYYY-MM-DD sync)

#auto-sync
```

Use strikethrough (`~~`) on the original meeting line and append the annotation. Do not delete the entry — preserving it maintains the audit trail of what was originally scheduled.

### 6. Discover Slack Activity

Use the Slack MCP tools to find conversations you participated in since `$SLACK_START_DATE`. Focus on **non-DM channels** (public channels, private channels, and group DMs — exclude 1:1 DMs).

**Step 1 — Find your messages:**

Use `slack_search_public_and_private` with `channel_types` set to `public_channel,private_channel,mpim`:

```
query: "from:<@YOUR_USER_ID> after:$SLACK_START_DATE"
channel_types: "public_channel,private_channel,mpim"
sort: "timestamp"
```

Use `slack_read_user_profile` (no arguments) to get your own user ID if needed.

**Step 2 — Find messages mentioning you:**

```
query: "to:<@YOUR_USER_ID> after:$SLACK_START_DATE"
channel_types: "public_channel,private_channel,mpim"
sort: "timestamp"
```

**Step 3 — Expand threads:**

For messages that are part of threads (indicated by `thread_ts`), use `slack_read_thread` to get the full conversation context. This helps generate meaningful summaries rather than one-sided fragments.

**Step 4 — Build interaction events:**

From the search results and thread context, build a list of Slack interaction events with:
- Timestamp (convert to local time for journal entries)
- Channel name
- Topic/subject of the conversation
- Key participants (resolve user IDs to names via `slack_search_users`)
- Brief summary of what was discussed or decided

**Step 5 — Match participants to people files:**

For each Slack user encountered in step 4:

1. **Grep for their Slack handle** in `people/`: search for `**Slack**: @<display-name>` across all people files. If found, the match is done — use that file's wiki-link format in journal entries.

2. **If no grep match**: use `slack_read_user_profile` to get the full profile (real name, title, team/department). Then grep `people/` for their real name in `**Name**:` lines.

3. **If a people file is found** (by either path): add a `- **Slack**: @<display-name>` line to the file header if not already present (so future syncs match on the fast grep path). If the profile reveals new information (title, team) not already recorded, update the header fields.

4. **If no people file exists** but the person had substantive interaction (not just reacting or being @mentioned in passing): note them in the sync summary as "new people seen in Slack" so the user can decide whether to create files. Don't auto-create people files from Slack alone — there's not enough context to pick the right filename convention (GitHub handle vs real name).

5. **For all matched people**: use their existing wiki-link format (`[[people/@handle|Name]]` or `[[people/first-last|Name]]`) in the generated journal entries.

**Grouping rules:**
- Multiple messages in the same thread should be a single entry at the earliest timestamp
- Separate threads/channels get separate timestamped entries
- Skip low-signal messages (simple reactions, one-word acknowledgements, bot notifications)
- If a conversation overlaps with a calendar meeting (same time window and participants), note the Slack context within the meeting entry rather than creating a duplicate

**Generate entries** in the standard auto-sync format:

```markdown
### HH:MM (auto-generated by brain-sync)

Slack: discussed **topic** in #channel with [[people/@handle|Name]] —
summary of the conversation or decision reached.

#auto-sync
```

If a conversation involved a notable decision or action item, call it out:

```markdown
### HH:MM (auto-generated by brain-sync)

Slack: **decision** in #channel — decided to <X> after discussion
with [[people/@handle|Name]]. Action: <next step>.

#auto-sync
```

### 7. Fill Gaps from Linear

Use the `linear` agent to query for recent activity. Use `$LINEAR_START_DATE` (from step 2) as the cutoff:

> List all issues assigned to me across all states that have had activity since $LINEAR_START_DATE. For each issue, include the minute-accurate timestamps (ISO 8601 with time, e.g. 2026-03-10T14:32:00Z) of: state changes (with from/to states), comments (with author), and assignment changes. I need the exact `createdAt`/`updatedAt` timestamps on each event, not just the issue-level `updatedAt`. If nothing has activity in this date range, just return an empty result — do not search outside the requested range. Only use the Linear MCP tools — do not try other data sources. If some of the requested data is not available in the MCP responses, note what's missing and skip it.

Use the returned issues and their event timestamps to supplement daily entries with Linear activity. Convert timestamps to local time for journal entry headings.

**Role assignment from Linear:** When creating or updating task files from Linear issues, set the `role` frontmatter field based on the assignee:
- Assigned to the current user → `role: owner`
- Assigned to someone else → `role: tracking`
- You're listed as a reviewer/subscriber but not assignee → `role: reviewer`
- If assignee is ambiguous or unset, omit `role` (defaults to `tracking`)

### 8. Update Task ↔ PR ↔ People Links

The `gh-pr-activity` output from step 4 already includes the fields needed for task linking:
- **`metadata.body`**: PR body text — extract Linear issue IDs from patterns like `Closes DEV-1234`, `Closes: DEV-1234`, `Fixes: DEV-1234`, `Refs: DEV-1234`
- **`metadata.user`**: PR author's GitHub login
- **`all_reviews`**: all reviewers with their final review state (deduplicated by login, `claude` bot excluded), each with `{login, state, submitted_at}`

For each Linear issue referenced by a PR:

1. **If a task file `tasks/DEV-XXXX.md` exists**: ensure it has:
   - `github:` field in frontmatter linking to the PR (format: `owner/repo#number`)
   - People links in the `## Related` section for the PR author (if not the current user) and reviewers (excluding the current user)
   - Use `[[people/@handle|Display Name]]` wiki-link format
   - Author link: `[[people/@handle|Name]] — authored PR #N`
   - Reviewer link: `[[people/@handle|Name]] — reviewed PR #N (STATE)`
   - Skip adding links that already exist

2. **If no task file exists**: create one following the brain-file skill format:
   - Set `status:` based on the Linear issue status from step 7 (map to: `active`, `done`, `blocked`, `cancelled`)
   - Set `role:` based on who authored the PR: `owner` if the current user authored it, `reviewer` if the current user only reviewed it, `tracking` otherwise. If Linear assignee info is available from step 7, prefer that over PR authorship.
   - Set `linear:` and `github:` frontmatter fields
   - Use the Linear issue title as the `##` heading
   - Add `[[projects/repo|repo]]` and people links in `## Related`
   - Add a `## Log` entry with the PR merge/creation date

For PRs **not** linked to any Linear issue, skip task creation (these are standalone chores/docs/deps).

**People file lookup**: read the `- **Name**:` line from each referenced person's file under `people/` to get their display name for wiki-links. If no people file exists for a reviewer, note it in the sync summary but don't create one automatically.

### 9. Generate/Update Daily Entries

For each day in the sync period, create or update the daily note with timestamped entries based on the interaction data gathered above.

**For days with no existing entry**, create a new file. **For days with an existing auto-sync block**, update it with any newly discovered activity.

Use approximate timestamps from the GitHub data (converted to local time) for each entry:

```markdown
### HH:MM (auto-generated by brain-sync)

Reviewed PR: [title](url) in [[projects/repo|repo]] (state) — brief summary
of the review or interaction based on comments/review body.

#auto-sync
```

**Grouping rules:**
- Multiple interactions on the same PR within a short window (< 30 min) should be a single entry at the earliest timestamp
- Separate PRs/issues get separate timestamped entries
- If no meaningful body/comments exist, write a one-line summary based on the PR title and interaction type

**Entry format by interaction type:**
- Authored PR: "Opened PR: [title](url) in [[projects/repo|repo]] — description from PR body"
- Reviewed PR: "Reviewed [title](url) in [[projects/repo|repo]] (APPROVED/CHANGES_REQUESTED/COMMENTED) — summary of review feedback"
- Commented on issue: "Commented on [title](url) in [[projects/repo|repo]] — summary of comment"
- Merged PR: "Merged [title](url) in [[projects/repo|repo]]"

Use `[[projects/repo|repo]]` wiki-links for repository/project names so daily entries connect to project nodes in the graph view.

### 10. Generate Weekly Rollup

If the current week has 5+ daily entries and no weekly rollup exists for this week:

Read all daily entries for the week and generate `journal/weekly/YYYY-Www.md`:

```markdown
## Summary
High-level summary of the week's work.

## Projects
- [[projects/<name>|Project]]: what happened this week

## Key PRs
- Authored: [#123 title](https://github.com/owner/repo/pull/123) in [[repo]], ...
- Reviewed: [#456 title](https://github.com/owner/repo/pull/456) in [[repo]], ...

Preserve full GitHub URLs from the daily entries so PRs remain clickable.

## Tasks
- Completed: list completed tasks
- In progress: list active tasks
- New: list newly created tasks

## People
- Notable interactions

## Key Decisions
- Link to any decision records filed this week

## Sources
- [[journal/daily/YYYY-MM-DD|YYYY-MM-DD]] (one entry per daily note included in this rollup)
```

The Sources section links back to every daily entry that was summarized, creating backlinks visible in Obsidian's graph view and backlinks panel.

### 11. Generate Monthly Rollup

If the current month has 3+ weekly rollups and no monthly rollup exists:

Read the weekly rollups and generate `journal/monthly/YYYY-MM.md` with similar structure but at monthly granularity. Include a `## Sources` section linking back to each weekly rollup:

```markdown
## Sources
- [[journal/weekly/YYYY-Www|Week Www]]
```

### 12. Audit Tasks

Read all files in `tasks/`:

- For each task with a `linear:` reference, check its status via the `linear` agent
- For each task with a `github:` reference, check its status via `gh issue view`
- If an external source shows the task as completed/closed but the brain shows it as `active`, flag it
- If a task has been `active` for more than 14 days without updates, flag it as potentially stale

Present flagged tasks with their title (the `##` heading from the task file) and ask the user which ones to update.

### 13. Commit

Write the sync state file using `$SYNC_TIMESTAMP` captured in step 2 (before any queries ran):

```bash
cat > ~/.local/share/second-brain/.brain-sync.json << EOF
{
  "github": {
    "last_query_at": "$SYNC_TIMESTAMP"
  },
  "linear": {
    "last_query_at": "$SYNC_TIMESTAMP"
  },
  "slack": {
    "last_query_at": "$SYNC_TIMESTAMP"
  }
}
EOF
```

Then commit all changes together:

```bash
jj -R ~/.local/share/second-brain commit -m "brain-sync: fill gaps and generate rollups"
```

## Output

After completing the sync, present a summary:

- Days synced: X/7
- New entries generated: N (GitHub: X, Calendar: X, Slack: X, Linear: X)
- Meetings reconciled: N moved/cancelled
- People files updated: N (with new Slack profile info)
- New people seen in Slack: list names (no file created — user to decide)
- Tasks linked: N new, N updated (PR/people links added)
- Rollups generated: weekly/monthly
- Tasks flagged: N stale, N out-of-sync
