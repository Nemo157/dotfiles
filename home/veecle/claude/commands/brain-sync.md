Synchronize the second brain at `~/.local/share/second-brain/` by detecting gaps in the journal, generating rollups, and auditing tasks.

## Workflow

Run these steps in order:

### 1. Check Brain State

```bash
ls ~/.local/share/second-brain/journal/daily/ | tail -14
```

Identify the last 7 days (from today backwards). Note which days have entries and which are missing.

### 2. Discover GitHub Activity

Query GitHub for all PRs and issues where `Nemo157` was involved within the `veecle` organization over the sync period:

```bash
# PRs involved in (covers authored, reviewed, commented, mentioned)
gh search prs --involves=Nemo157 --owner=veecle --updated='>=$START_DATE' \
  --json number,title,repository,url,state,updatedAt,createdAt

# Issues involved in
gh search issues --involves=Nemo157 --owner=veecle --updated='>=$START_DATE' \
  --json number,title,repository,url,state,updatedAt,createdAt
```

Use `$START_DATE` as the earliest missing day (or 7 days ago if updating existing entries).

### 3. Inspect Each PR/Issue for Interaction Timestamps

For each PR/issue found, fetch the detailed interaction history to determine when `Nemo157` actually interacted with it:

```bash
# PR metadata (authorship, creation, merge)
gh api repos/OWNER/REPO/pulls/NUMBER \
  --jq '{created_at, merged_at, user: .user.login, body}'

# Reviews by Nemo157
gh api repos/OWNER/REPO/pulls/NUMBER/reviews \
  --jq '.[] | select(.user.login == "Nemo157") | {submitted_at, state}'

# Review comments
gh api repos/OWNER/REPO/pulls/NUMBER/comments \
  --jq '.[] | select(.user.login == "Nemo157") | {created_at, body}'

# Issue comments
gh api repos/OWNER/REPO/issues/NUMBER/comments \
  --jq '.[] | select(.user.login == "Nemo157") | {created_at, body}'

# Commits authored (for authored PRs)
gh api repos/OWNER/REPO/pulls/NUMBER/commits \
  --jq '.[] | select(.author.login == "Nemo157" or .committer.login == "Nemo157") | {date: .commit.author.date, message: .commit.message}'
```

From these, build a list of interaction events with timestamps:
- **Authored PR**: use the PR `created_at`
- **Reviewed PR**: use each review's `submitted_at`
- **Commented**: use each comment's `created_at`
- **Committed**: use the commit date
- **Merged own PR**: use `merged_at` if the user is the author

Group these events by date (in local time) to assign them to the correct daily journal.

### 4. Discover Calendar Events

Use the Google Calendar MCP tools to find meetings for each day in the sync period:

- Use `list-calendars` to identify the work calendar
- Use `list-events` for each day in the sync period, filtering to the work calendar
- For each meeting: extract time, title, attendees

**Attendee formatting:**
- If attendees are only mailing lists or group addresses, list participants as "dev team" instead of expanding the list names
- If a meeting has both mailing lists and named individuals, use "dev team" for the lists and list the named individuals separately
- Only list named individuals when they are explicitly in the attendee list (not behind a mailing list)

Generate `### HH:MM (auto-generated by brain-sync)` entries with `#auto-sync` tag:

```markdown
### HH:MM (auto-generated by brain-sync)

Meeting: **title** with dev team

#auto-sync
```

### 5. Fill Gaps from Linear

If the `linear` CLI is available, query for recent activity:

```bash
linear issue list --all-states --assignee wim --no-pager 2>/dev/null || true
```

Note: Linear may not be available in all contexts. Skip silently if the command fails.

### 6. Generate/Update Daily Entries

For each day in the sync period, create or update the daily note with timestamped entries based on the interaction data gathered above.

**For days with no existing entry**, create a new file. **For days with an existing auto-sync block**, update it with any newly discovered activity.

Use approximate timestamps from the GitHub data (converted to local time) for each entry:

```markdown
### HH:MM (auto-generated by brain-sync)

Reviewed PR: [title](url) in [[projects/repo|repo]] (state) — brief summary
of the review or interaction based on comments/review body.

#auto-sync
```

**Grouping rules:**
- Multiple interactions on the same PR within a short window (< 30 min) should be a single entry at the earliest timestamp
- Separate PRs/issues get separate timestamped entries
- If no meaningful body/comments exist, write a one-line summary based on the PR title and interaction type

**Entry format by interaction type:**
- Authored PR: "Opened PR: [title](url) in [[projects/repo|repo]] — description from PR body"
- Reviewed PR: "Reviewed [title](url) in [[projects/repo|repo]] (APPROVED/CHANGES_REQUESTED/COMMENTED) — summary of review feedback"
- Commented on issue: "Commented on [title](url) in [[projects/repo|repo]] — summary of comment"
- Merged PR: "Merged [title](url) in [[projects/repo|repo]]"

Use `[[projects/repo|repo]]` wiki-links for repository/project names so daily entries connect to project nodes in the graph view.

### 7. Generate Weekly Rollup

If the current week has 5+ daily entries and no weekly rollup exists for this week:

Read all daily entries for the week and generate `journal/weekly/YYYY-Www.md`:

```markdown
## Summary
High-level summary of the week's work.

## Projects
- [[projects/<name>|Project]]: what happened this week

## Key PRs
- Authored: [#123 title](https://github.com/owner/repo/pull/123) in [[repo]], ...
- Reviewed: [#456 title](https://github.com/owner/repo/pull/456) in [[repo]], ...

Preserve full GitHub URLs from the daily entries so PRs remain clickable.

## Tasks
- Completed: list completed tasks
- In progress: list active tasks
- New: list newly created tasks

## People
- Notable interactions

## Key Decisions
- Link to any decision records filed this week

## Sources
- [[journal/daily/YYYY-MM-DD|YYYY-MM-DD]] (one entry per daily note included in this rollup)
```

The Sources section links back to every daily entry that was summarized, creating backlinks visible in Obsidian's graph view and backlinks panel.

### 8. Generate Monthly Rollup

If the current month has 3+ weekly rollups and no monthly rollup exists:

Read the weekly rollups and generate `journal/monthly/YYYY-MM.md` with similar structure but at monthly granularity. Include a `## Sources` section linking back to each weekly rollup:

```markdown
## Sources
- [[journal/weekly/YYYY-Www|Week Www]]
```

### 9. Audit Tasks

Read all files in `tasks/`:

- For each task with a `linear:` reference, check its status via `linear issue view`
- For each task with a `github:` reference, check its status via `gh issue view`
- If an external source shows the task as completed/closed but the brain shows it as `active`, flag it
- If a task has been `active` for more than 14 days without updates, flag it as potentially stale

Present flagged tasks and ask the user which ones to update.

### 10. Commit

After all changes:

```bash
jj -R ~/.local/share/second-brain commit -m "brain-sync: fill gaps and generate rollups"
```

## Output

After completing the sync, present a summary:

- Days synced: X/7
- New entries generated: N
- Rollups generated: weekly/monthly
- Tasks flagged: N stale, N out-of-sync
